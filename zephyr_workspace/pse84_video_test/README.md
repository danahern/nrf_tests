# pse84_video_test

24 fps embedded animation playback demo for the PSE84 GFXSS display driver.
Loops a 125-frame, 5.208 s animation of a fuzzy round character on the
Waveshare 4.3" MIPI-DSI panel (800x480, RGB565).

## What it does

- Frames are pre-baked to raw RGB565 at **240x144** resolution and embedded
  as a single 8.64 MB const blob in flash via
  `generate_inc_file_for_target(app src/frames.bin ...)`.
- At runtime `src/main.c` pixel-doubles each frame **3x** to **720x432** and
  renders it centered on the 800x480 panel with a 40 px left/right /
  24 px top/bottom black border.
- Timing is paced by `k_uptime_get()` against a 42 ms frame period.
- The M55 display driver is `infineon,pse84-gfxss` (from the upstream-
  branch display driver series); the framebuffer lives in SOCMEM at
  `socmem_fb` (1 MB region at 0x26240000).

## Why 240x144 -> 3x upscale instead of full 800x480 native

The kit's QSPI flash (the one Zephyr currently maps) is only 16 MB and the
existing boot chain (storage + M33 secure + M33 NS + M55 XIP) leaves ~9 MB
free past `m55_xip`. A full 125-frame 800x480 RGB565 loop would need
93.8 MB — more than the QSPI can hold. The board has a 128 MB Octal SEMPER
flash available on SMIF0 CS0 but Zephyr doesn't yet wire it up, and the
octal and quad flashes share IO lines so only one can be electrically
active at a time.

See `PSE84_OCTAL_FLASH_ENABLEMENT_PLAN.md` (in the outer
`zephyr_workspace/` dir, generated by a separate planning pass) for the
roadmap to enable the full 128 MB octal flash. Once that lands this demo
can switch to full 800x480 native frames without any playback code
changes.

## Memory layout

- `src/frames.bin` — 8,640,000 bytes (125 * 240 * 144 * 2)
- `boards/kit_pse84_eval_pse846gps2dbzc4a_m55.overlay` extends the
  `m55_xip` partition from its default 2 MB to 10 MB with a local
  `&m55_xip { reg = <0x60500000 DT_SIZE_M(10)>; };` override. This is NOT
  upstreamed — it's app-local and leaves other PSE84 apps' 2 MB partition
  unchanged.
- The upstream `soc/infineon/edge/pse84/security_config/pse84_s_protection.c`
  had a 3 MB M55 SMIF0 MPC region cap that this demo needed extended to
  11 MB. That one-line bump IS upstreamed (see commit
  `soc: infineon: pse84: extend M55 SMIF0 MPC region to 11 MB` on the
  `pse84-gfxss-display-driver` branch).

## Regenerating frames from the source video

```
ffmpeg -i ~/Downloads/a-looping-3-second-animation-of-a-big-round-fuzzy-\ \(1\).mp4 \
    -vf "scale=258:144:flags=lanczos,crop=240:144" \
    -pix_fmt rgb565le -f rawvideo src/frames.bin
```

Expected output: `8640000 bytes` (125 frames * 240 * 144 * 2 bpp).

## Build & flash

```
west build -b kit_pse84_eval/pse846gps2dbzc4a/m55 --sysbuild \
    ../pse84_video_test -p always -d ../pse84_video_test/build_m55

west flash -d ../pse84_video_test/build_m55 --domain enable_cm55
west flash -d ../pse84_video_test/build_m55 --domain pse84_video_test
```

The M55 flash takes ~73 s because we're writing ~8.7 MB to a Quad SPI
NOR at ~120 KB/s. Reflashing the M33 companion is only needed once
after the MPC fix is in place.

## Known issues

- Halting CM33 via openocd during playback sometimes disrupts
  `Cy_SysEnableCM55` state and causes subsequent resumes to fail. Power
  cycle or full `reset run` to recover.
- M33 companion still reaches the Zephyr fatal-error spin loop after
  enabling CM55 (pre-existing latent bug in `pse84_boot.c`, unrelated to
  this demo, documented in project memory under
  `project_pse84_m33_hardfault.md`).
